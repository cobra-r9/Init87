# 1.1 — Integer Types
## The Complete Truth About `char`, `short`, `int`, `long`, `long long`



## Start Here — What an Integer Type Actually Is

When you write:

```c
int x = 42;
```

Three decisions are made by the compiler, silently, instantly:

**Decision 1 — Size:** How many bytes to reserve for `x`.

**Decision 2 — Alignment:** What address boundary `x` must start on.

**Decision 3 — Instruction selection:** Which load/store/arithmetic instructions to emit when you use `x`.

The type `int` drives all three. Not the value 42. The type.

Change `int` to `char` and all three change. The value 42 fits in either. The machine code is different.



## The Five Base Integer Types

```c
char        // smallest addressable integer unit
short       // at least 16 bits
int         // natural word size, at least 16 bits
long        // at least 32 bits
long long   // at least 64 bits (C99)
```

Each has a signed and unsigned variant:

```c
signed char        unsigned char
signed short       unsigned short
signed int         unsigned int
signed long        unsigned long
signed long long   unsigned long long
```

`signed int` and `int` are the same thing. `int` is signed by default.

`char` is the exception — whether plain `char` is signed or unsigned is **implementation-defined.** On x86 Linux with GCC it's signed. On ARM it's often unsigned. This is not a minor quirk. It is a real source of bugs when `char` values are compared to EOF (-1) or used in arithmetic.

**Never use plain `char` for arithmetic. Ever.**

Use `unsigned char` for raw bytes. Use `signed char` if you explicitly need a small signed integer. Use `char` only for actual text characters, and even then be careful.



## The Actual Sizes — On Your Machine, Right Now

The C standard guarantees minimums. Your platform delivers specifics.

On **x86-64 Linux with GCC** (your machine):

```
TYPE            SIZE        ALIGNMENT     RANGE (signed)
──────────────────────────────────────────────────────────────────────
char            1 byte      1 byte        -128 to 127
short           2 bytes     2 bytes       -32,768 to 32,767
int             4 bytes     4 bytes       -2,147,483,648 to 2,147,483,647
long            8 bytes     8 bytes       -9,223,372,036,854,775,808 to ...
long long       8 bytes     8 bytes       same as long on this platform
```

```
TYPE            SIZE        RANGE (unsigned)
──────────────────────────────────────────────────────────────────────
unsigned char   1 byte      0 to 255
unsigned short  2 bytes     0 to 65,535
unsigned int    4 bytes     0 to 4,294,967,295
unsigned long   8 bytes     0 to 18,446,744,073,709,551,615
unsigned long long  8 bytes same as unsigned long on this platform
```

Verify it yourself — always:

```c
#include <stdio.h>

int main(void) {
    printf("char      : %zu\n", sizeof(char));
    printf("short     : %zu\n", sizeof(short));
    printf("int       : %zu\n", sizeof(int));
    printf("long      : %zu\n", sizeof(long));
    printf("long long : %zu\n", sizeof(long long));
    return 0;
}
```
> [!IMPORTANT]
> `sizeof` returns type `size_t` — print with `%zu`. Using `%d` for sizeof is a bug — it's unsigned and may be 64-bit.



## The Platform Betrayal — Why Sizes Lie

On 64-bit Linux: `sizeof(long)` = 8.
On 64-bit Windows: `sizeof(long)` = 4.

Same hardware. Different OS. Different size. No warning. No error.

This is not hypothetical. This has broken real code. Security-critical code. Code where "I'll use `long` to hold a pointer" worked on Linux and silently truncated addresses on Windows.

The LP64 vs LLP64 data models:

```
MODEL       char  short  int   long  long long  pointer
─────────────────────────────────────────────────────────
ILP32       1     2      4     4     8           4      (32-bit)
LP64        1     2      4     8     8           8      (64-bit Linux/macOS)
LLP64       1     2      4     4     8           8      (64-bit Windows)
ILP64       1     2      8     8     8           8      (rare)
```

Linux uses LP64. Windows uses LLP64. `long` is the only type that differs.

The consequence: any code that uses `long` to mean "pointer-sized integer" is **wrong on Windows.** Use `intptr_t` instead. That's what it exists for.



## Two's Complement — The Full Story

All modern CPUs represent signed integers in **two's complement.**

C11 and earlier technically allowed ones' complement and sign-magnitude. Nobody used them. C20 finally mandated two's complement for signed integers. The world caught up to reality.

**Why two's complement?**

Take 8 bits. The 256 possible bit patterns need to represent 256 values.

Sign-magnitude wastes one pattern: both `00000000` (+0) and `10000000` (-0) exist. Two zeros. Arithmetic gets complicated.

Ones' complement also has two zeros. Also complicated.

Two's complement: no wasted pattern. One zero. Arithmetic is clean.

The bit pattern for -1 in an N-bit two's complement integer is all 1s:

```
 8-bit: 11111111
16-bit: 1111111111111111
32-bit: 11111111111111111111111111111111
```

The negation rule: flip all bits, add 1.

```
 42 = 00101010
-42 = 11010101 + 1 = 11010110
```

Verify: 42 + (-42) = 00101010 + 11010110 = 100000000. The carry out of the 8th bit is discarded. Result: 00000000 = 0. Correct.

**The profound consequence:**

```c
unsigned int u = 42;
int          s = 42;
```

Both store `00000000 00000000 00000000 00101010` in memory.

```c
unsigned int u = (unsigned int)-1;  // 0xFFFFFFFF = 4294967295
int          s = -1;                // 0xFFFFFFFF = -1
```

Same bit pattern. Different interpretation. The CPU uses the same ADD instruction for both. Only comparisons and shifts differ between signed and unsigned.



## The Ranges — Derived From First Principles

Don't memorize these. Derive them.

**8-bit signed (char):**
- High bit = sign. Remaining 7 bits = magnitude.
- Most positive: `01111111` = 2^7 - 1 = **127**
- Most negative: `10000000` = -2^7 = **-128**
- Range: -128 to 127

**8-bit unsigned:**
- All 8 bits = magnitude
- Maximum: `11111111` = 2^8 - 1 = **255**
- Range: 0 to 255

**N-bit signed:** -2^(N-1) to 2^(N-1) - 1

**N-bit unsigned:** 0 to 2^N - 1

Note the asymmetry: there are more negative values than positive. -128 exists. +128 doesn't fit in 8-bit signed. This asymmetry causes bugs:

```c
signed char x = -128;
x = -x;   // attempting to negate -128
           // +128 doesn't fit in signed char
           // overflow — undefined behavior
           // in practice on x86: still -128
```



## Signed Overflow — Undefined Behavior, Not Wrapping

This is where many C programmers are wrong.

**Unsigned overflow is defined.** It wraps modulo 2^N. Always.

```c
unsigned char x = 255;
x++;   // x = 0. Defined. Guaranteed.
```

**Signed overflow is undefined behavior.** The C standard says the result is undefined. Not "wraps." Not "implementation-defined." **Undefined.**

```c
signed char x = 127;
x++;   // undefined behavior
       // on x86 in practice: -128
       // but the compiler is allowed to assume this never happens
       // and optimize accordingly
```

"The compiler is allowed to assume this never happens" is the dangerous part.

GCC at -O2 will see:

```c
for (signed char i = 0; i < 200; i++) {
    // ...
}
```

And reason: "A signed char can never reach 200 without overflowing. Signed overflow is UB. Therefore I can assume the loop runs forever." It may emit an infinite loop. Legally, per the C standard.

This isn't theoretical. This class of bug has caused real security vulnerabilities. The compiler optimizes away an overflow check because the optimizer knows signed overflow is UB:

```c
// Intended: check for overflow before multiplication
if (a + b < a) {
    // overflow!
}
```

The compiler sees: if `a + b` overflows, that's UB, so I can assume it doesn't happen, so `a + b < a` is always false, so remove the branch entirely. Your overflow check is gone. Silent. No warning.

**Use unsigned arithmetic for overflow-sensitive code. Or use `-fwrapv` which makes signed overflow defined as wrapping (at a performance cost).**



## Integer Promotion — The Invisible Conversion

C has a rule that surprises most programmers:

> **In any expression, values of types smaller than `int` are automatically promoted to `int` before the operation.**

This means `char` and `short` almost never participate in arithmetic as their actual types. They get promoted.

```c
char a = 100;
char b = 100;
char c = a + b;   // what happens?
```

Step by step:

1. `a` is promoted from `char` to `int` → 100
2. `b` is promoted from `char` to `int` → 100
3. Addition: `int + int` = 200 (no overflow — fits in int)
4. Result 200 assigned to `char c` — truncated to 8 bits → -56

The addition happened in `int` arithmetic. The truncation happened on assignment. The intermediate result of 200 — which doesn't fit in `char` — was fine during computation.

Without promotion:
- `char + char` would overflow at 8 bits
- 100 + 100 = 200, which doesn't fit in signed char (max 127)

With promotion:
- `int + int` = 200, fine
- Truncated to char on assignment: wraps to -56

Both are "wrong" depending on what you wanted. The point is: **you must understand promotion to know what will happen.**

Promotion rules:

```
char        → int  (or unsigned int if plain char is unsigned and fits)
short       → int
unsigned char → int (value always fits)
unsigned short → int (on 64-bit where int is 32-bit, always fits)
```

`int` and larger types are not promoted further.



## Usual Arithmetic Conversions — When Types Mix

When two different types appear in one expression, C converts them to a common type:

```
1. If either is long double → both become long double
2. Else if either is double → both become double
3. Else if either is float → both become float
4. Otherwise, apply integer promotions, then:
   - If same type → done
   - If both signed or both unsigned → smaller converts to larger
   - If unsigned rank ≥ signed rank → signed converts to unsigned
   - If signed can represent all values of unsigned → unsigned to signed
   - Else → both convert to unsigned version of the signed type
```

Rule 4's third case is the killer:

```c
int           s = -1;
unsigned int  u = 1;

if (s < u) {
    printf("yes\n");   // you expect this
} else {
    printf("no\n");    // this prints
}
```

`-1 < 1` should be true. But `s` is converted to `unsigned int`. -1 as unsigned int = 4,294,967,295. 4,294,967,295 < 1 is false.

```c
// Classic vulnerability:
size_t len = get_length_from_network();  // attacker controls this
if (len < 1024) {                        // seems safe
    char buf[1024];
    memcpy(buf, data, len);              // but len could be SIZE_MAX
}
```

If `len` comes from an attacker and is `SIZE_MAX` (0xFFFFFFFFFFFFFFFF), the comparison `len < 1024` is false — correctly rejected.

But consider this version:

```c
int len = get_length_from_network();    // signed int!
if (len < 1024) {
    char buf[1024];
    memcpy(buf, data, len);
}
```

Attacker sends -1. `len = -1`. `-1 < 1024` is true. `memcpy(buf, data, -1)`. But `memcpy`'s third argument is `size_t` — unsigned. -1 converts to SIZE_MAX. You just copied gigabytes over your stack.

**This exact class of vulnerability — signed/unsigned mismatch in a length check — appears in the CVE database hundreds of times.**



## What the Compiler Actually Emits — Per Type

Let's make this concrete. This C:

```c
int add_int(int a, int b)       { return a + b; }
long add_long(long a, long b)   { return a + b; }
char add_char(char a, char b)   { return a + b; }
```

Compiles to (`gcc -O2 -S`):

```asm
add_int:
    lea  eax, [rdi + rsi]    ; 32-bit add, result in EAX
    ret

add_long:
    lea  rax, [rdi + rsi]    ; 64-bit add, result in RAX
    ret

add_char:
    add  edi, esi            ; 32-bit add (args promoted to int)
    movsx eax, dil           ; sign-extend low byte to 32-bit return
    ret
```

`add_int` uses 32-bit registers. `add_long` uses 64-bit registers. `add_char` promotes to 32-bit, adds, then sign-extends back to byte — that's the promotion happening in the emitted code.

Same source structure. Different instructions. The type was the instruction.



## Integer Ranks — The Formal Ordering

C defines a **rank** for integer types. Higher rank wins in conversions:

```
_Bool < char < short < int < long < long long
```

Unsigned types have the same rank as their signed counterpart.

This rank is why `int + long` gives `long` — `long` outranks `int`. It's why `int + unsigned int` gives `unsigned int` on a platform where they're the same size — unsigned beats signed at the same rank.



## The Actual Sizes — Reading `<limits.h>`

`<limits.h>` contains the actual constants for your platform:

```c
#include <limits.h>
#include <stdio.h>

int main(void) {
    printf("CHAR_BIT   = %d\n",  CHAR_BIT);    // bits per char (8)
    printf("CHAR_MIN   = %d\n",  CHAR_MIN);    // -128 or 0
    printf("CHAR_MAX   = %d\n",  CHAR_MAX);    // 127 or 255
    printf("SHRT_MIN   = %d\n",  SHRT_MIN);    // -32768
    printf("SHRT_MAX   = %d\n",  SHRT_MAX);    // 32767
    printf("INT_MIN    = %d\n",  INT_MIN);     // -2147483648
    printf("INT_MAX    = %d\n",  INT_MAX);     // 2147483647
    printf("LONG_MIN   = %ld\n", LONG_MIN);    // -9223372036854775808
    printf("LONG_MAX   = %ld\n", LONG_MAX);
    printf("LLONG_MAX  = %lld\n", LLONG_MAX);
    printf("UINT_MAX   = %u\n",  UINT_MAX);    // 4294967295
    printf("ULONG_MAX  = %lu\n", ULONG_MAX);
    return 0;
}
```

Run this. These are not hypothetical. These are the actual values your compiler will use for every int expression you write.



## `sizeof` — The Compile-Time Oracle

`sizeof(T)` returns the size of type T in bytes. It's evaluated **at compile time** (except VLAs). The result type is `size_t` — always unsigned, always pointer-sized.

```c
sizeof(char)        // always 1, by definition
sizeof(short)       // 2 on all common platforms
sizeof(int)         // 4 on all common modern platforms
sizeof(long)        // 4 on Windows 64-bit, 8 on Linux 64-bit
sizeof(long long)   // 8 everywhere
sizeof(void *)      // 4 on 32-bit, 8 on 64-bit
```

`sizeof(char)` is defined to be 1. This does not mean char is 1 octet (8 bits) everywhere — `CHAR_BIT` tells you the actual bit width. On some DSPs, `CHAR_BIT` = 16 or 32 and `sizeof(char)` is still 1, but a char is 16 or 32 bits wide.

You will never target such a platform. But you will understand why `sizeof(char) == 1` and "one byte" are not identical statements.



## The `_Bool` Type

C99 added `_Bool`. Include `<stdbool.h>` for the `bool`/`true`/`false` aliases.

```c
#include <stdbool.h>
bool flag = true;    // same as _Bool flag = 1
```

`_Bool` is 1 byte. Any non-zero value assigned to it becomes 1. Zero becomes 0. It's not special in arithmetic — it promotes to `int` like everything else.

```c
_Bool b = 42;    // b = 1, not 42
_Bool b = -5;    // b = 1
_Bool b =  0;    // b = 0
```



## Common Mistakes — The Ones That Bite

**Mistake 1 — Assuming `int` is 64 bits:**
```c
int x = 3000000000;   // doesn't fit in 32-bit signed int
                      // INT_MAX = 2147483647
                      // this is overflow at initialization — UB
```

**Mistake 2 — Mixing signed and unsigned:**
```c
unsigned int len = get_len();
int i;
for (i = len - 1; i >= 0; i--) {  // if len = 0, len-1 = UINT_MAX
    // loops 4 billion times
}
```

**Mistake 3 — Using `char` for bytes:**
```c
char c = getchar();
if (c == EOF) { ... }   // EOF is -1
                        // if char is unsigned, c is never -1
                        // use int, not char
```

**Mistake 4 — Shift overflow:**
```c
int x = 1 << 31;   // shifting into sign bit — UB in C
                   // use: 1u << 31  (unsigned, defined)
```

**Mistake 5 — Sizeof in comparisons:**
```c
if (sizeof(x) == -1) { ... }  // sizeof returns unsigned size_t
                               // -1 converted to size_t = SIZE_MAX
                               // comparison is always false
                               // compiler may warn, may not
```



## Summary — What You Own After 1.1

```
5 base types     char short int long long long
Each has         signed (default) and unsigned variant
Sizes            platform-dependent — check sizeof and <limits.h>
LP64 vs LLP64    long is 8 bytes Linux, 4 bytes Windows
Two's complement flip+1, asymmetric range, -2^(N-1) to 2^(N-1)-1
Signed overflow  undefined behavior — compiler exploits this
Unsigned overflow wraps modulo 2^N — defined
Promotion        char/short → int before arithmetic, always
Mixed arithmetic signed+unsigned → unsigned, dangerous
sizeof           compile-time, returns size_t, always unsigned
char signedness  implementation-defined — never assume
```

One sentence:

> **Integer types are compile-time instructions to the compiler specifying exactly how many bytes to reserve, how to align them, and which machine instructions to emit — and the silent rules of promotion, conversion, and undefined overflow are where security vulnerabilities are born.**
